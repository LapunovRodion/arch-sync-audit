# Правила агента для этого репозитория

Этот репозиторий является учебным проектом для самостоятельного изучения Rust.

## Основные принципы

- Главная цель проекта: обучение владельца репозитория написанию кода на Rust.
- Агент не должен добавлять, изменять или удалять код в проекте без прямой просьбы пользователя.
- Если пользователь просит объяснить тему, ошибку, подход или фрагмент кода, агент должен сначала объяснять, а не сразу менять файлы.
- Объяснения должны быть достаточно подробными, особенно когда речь идет о Rust, архитектуре, ошибках компиляции, типах, владении, заимствовании, lifetimes, модулях, тестах и инструментах.
- Предпочтительно помогать пользователю писать код руками: давать пошаговые инструкции, небольшие фрагменты, подсказки и объяснения вместо полной вставки готового решения.
- Полный код можно давать только если пользователь явно попросил полный пример или изменение файла.

## Работа с кодом

- Перед изменением кода агент должен кратко объяснить, что именно собирается изменить и зачем.
- Если задача неоднозначна, агент должен сначала задать уточняющие вопросы.
- Если пользователь просит реализовать изменение, агент должен делать минимальные корректные правки.
- Агент не должен расширять задачу, добавлять лишние абстракции или переписывать несвязанный код без необходимости.
- После изменений агент должен запускать только релевантные проверки, например `cargo check`, `cargo test` или конкретные тесты, если это применимо.

## Формат помощи

- Отвечать на русском языке в техническом стиле.
- Для учебных задач сначала объяснять идею, затем давать маленькие шаги для самостоятельного написания.
- Если есть несколько вариантов решения, кратко перечислить их, объяснить риски и рекомендовать один вариант.
- Если пользователь просит подсказку, не раскрывать сразу всё решение.
- Если пользователь просит проверить написанный код, давать ревью с объяснением ошибок и возможных улучшений.

## Документация

- Агент должен вести подробную документацию по проекту, архитектурным решениям, командам, настройкам и важным выводам.
- Всю проектную документацию пишет и поддерживает агент, если пользователь явно просит задокументировать тему, решение или результат работы.
- Документация должна быть понятной для самостоятельного обучения: объяснять не только что сделано, но и зачем это нужно.

## Git и файлы

- Не создавать коммиты, не пушить изменения и не открывать pull request без явной команды пользователя.
- Не трогать чужие или неожиданные изменения в рабочем дереве.
- Не менять файлы, не относящиеся к текущей задаче.
