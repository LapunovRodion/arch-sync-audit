# Заметки разработки

Этот документ ведется как учебный журнал проекта. В него нужно записывать важные решения, причины этих решений и изученные Rust-темы.

## Текущие решения

### Проект является учебным

Главная цель - научиться писать на Rust. Поэтому разработка идет маленькими шагами, с объяснением тем и ручным написанием кода владельцем проекта.

### Агент не меняет код без просьбы

Код проекта изменяется агентом только по прямой просьбе пользователя. По умолчанию агент объясняет, предлагает шаги и помогает писать код руками.

### Сначала чистая логика, потом системные команды

Сравнение профилей и генерация плана должны быть отделены от чтения файлов и запуска `pacman`. Это упростит тестирование и обучение.

### `plan` безопасен

Команда `plan` только печатает действия. Она не выполняет установку пакетов, не меняет shell и не включает сервисы. Для AUR и systemd services она печатает комментарии, чтобы пользователь сознательно выполнил действия вручную.

### Конфиги в MVP проверяются только по путям

На первом этапе программа не читает содержимое конфигов и не копирует dotfiles. Она только проверяет, существуют ли важные пути.

### `systemctl --user` является частично надежным источником

Пользовательский systemd может быть недоступен в некоторых окружениях. Поэтому ошибка `systemctl --user` не ломает `export`, `check` и `plan`: программа печатает warning в stderr и продолжает работу с пустым списком `user_services`.

### Вывод отделен от сравнения

`compare.rs` возвращает данные, а не готовый текст. Отчет для `check` формируется в `report.rs`, а безопасный план для `plan` формируется в `plan.rs`. Это сохраняет сравнение тестируемым и не привязывает его к конкретному CLI-выводу.

## Реализованные учебные этапы

- минимальный CLI на `clap`;
- модель `SystemProfile`;
- сериализация и чтение TOML;
- чистое сравнение профилей;
- unit-тесты для сравнения;
- сбор `$SHELL`;
- запуск внешних команд через `std::process::Command`;
- экспорт pacman/AUR-пакетов;
- реальный `check` по пакетам и shell;
- безопасный `plan`;
- проверка config paths по факту существования;
- сбор enabled systemd services;
- структурированный отчет `check`.

По мере разработки проекта будут изучаться:

- `cargo` и структура проекта;
- `fn main()`;
- `struct`;
- `enum`;
- `match`;
- `Vec<T>`;
- `Option<T>`;
- `Result<T, E>`;
- оператор `?`;
- владение и заимствование;
- `String` и `&str`;
- модули;
- `serde`;
- работа с файлами;
- запуск внешних команд;
- тесты;
- `HashSet`;
- `PathBuf`;
- парсинг строкового вывода команд;
- разделение чистой логики и side effects.
